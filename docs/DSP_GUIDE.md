# DSP (Data Structure Protocol) Guide

**Last Updated**: 2026-03-25

DSP is a filesystem-based codebase graph that maps all source entities (files, modules, functions) and their import relationships. It provides structural memory for LLM agents, enabling them to understand code dependencies, detect cycles, and navigate the codebase before making changes.

---

## Overview

The DSP graph for RateShift currently contains:
- **474 entities** (436 source + 38 external)
- **940+ import relationships**
- **1 real cycle** (`alert_service` <-> `alert_renderer`)

All data lives in the `.dsp/` directory at the project root. The graph is rebuilt from scratch using the auto-bootstrap script and can be queried via `dsp-cli.py`.

---

## Critical: Wipe-Before-Rebuild Pattern

The bootstrap script **appends** entities -- it does not replace existing ones. Always wipe the `.dsp/` directory before rebuilding:

```bash
rm -rf .dsp && python3 dsp-cli.py --root . init && python3 scripts/dsp_auto_bootstrap.py
```

Failing to wipe first will result in duplicate entities and broken relationships.

## Directory Structure

```
.dsp/
├── TOC                          # Table of contents (root entity list)
├── uid_map.json                 # source_path -> UID mapping (for lookups)
├── obj-a1b2c3d4/                # Entity directory (object)
│   ├── description              # source, kind, purpose fields
│   ├── imports                  # List of imported entity UIDs
│   ├── shared                   # List of shared/exported entity UIDs
│   └── exports/                 # Reverse references (who imports this)
│       ├── obj-e5f6g7h8         # Direct importer with "why" text
│       └── func-i9j0k1l2/      # Shared export subdirectory
│           ├── description      # What is shared
│           └── obj-m3n4o5p6     # Importer with "why" text
└── func-q7r8s9t0/              # Entity directory (function)
    └── ...
```

### Entity Types

| UID Prefix | Kind | Description |
|------------|------|-------------|
| `obj-*` | `object` | Source file (module, component, service) |
| `obj-*` | `external` | External package (fastapi, react, etc.) |
| `func-*` | `function` | Individual function or method |

### Description File Format

```
source: backend/services/price_service.py
kind: object
purpose: Business logic service for price data operations
```

## `uid_map.json`

A JSON mapping from source file paths to entity UIDs:

```json
{
  "backend/services/price_service.py": "obj-a1b2c3d4",
  "frontend/components/PriceChart.tsx": "obj-e5f6g7h8",
  "fastapi": "obj-i9j0k1l2"
}
```

This file is generated by the auto-bootstrap script and is the primary lookup mechanism for finding entity UIDs from file paths.

## CLI Commands

All commands use the pattern:
```bash
python3 dsp-cli.py --root . <command> [args]
```

### `init`

Initialize the `.dsp/` directory structure.

```bash
python3 dsp-cli.py --root . init
```

### `search`

Full-text search across all entity descriptions and export names. Case-insensitive substring matching.

```bash
python3 dsp-cli.py --root . search "price_service"
# Output: obj-a1b2c3d4  [source] backend/services/price_service.py
```

### `get-recipients`

Find all entities that import a given entity (reverse dependency lookup). Essential before refactoring -- shows what will break.

```bash
python3 dsp-cli.py --root . get-recipients obj-a1b2c3d4
# Output: list of (uid, why) tuples showing all importers
```

### `get-children`

Walk the import tree downward from an entity. Use `--depth` to control traversal depth (default: 1, use `inf` for full tree).

```bash
python3 dsp-cli.py --root . get-children obj-a1b2c3d4 --depth 2
# Output: tree view showing imports and their imports
```

### `get-parents`

Walk the import tree upward (who imports this, and who imports them).

```bash
python3 dsp-cli.py --root . get-parents obj-a1b2c3d4 --depth 2
```

### `find-by-source`

Find entity UIDs by source file path. Useful when you know the file but not the UID.

```bash
python3 dsp-cli.py --root . find-by-source backend/services/price_service.py
# Output: obj-a1b2c3d4
```

### `detect-cycles`

Find circular dependencies in the import graph.

```bash
python3 dsp-cli.py --root . detect-cycles
# Output: cycle 1: obj-abc -> obj-def -> obj-abc
```

### `get-stats`

Print graph-level statistics.

```bash
python3 dsp-cli.py --root . get-stats
# Output:
# entities:  474
#   objects:   398
#   functions: 38
#   external:  38
# imports:   940
# shared:    0
# cycles:    1
# orphans:   12
```

### Additional Commands

| Command | Description |
|---------|-------------|
| `create-object` | Create a new object entity with source, purpose, kind |
| `create-function` | Create a new function entity (optionally owned by an object) |
| `create-shared` | Register shared/exported entities |
| `add-import` | Add an import relationship between two entities |
| `update-description` | Update entity description fields |
| `update-import-why` | Update the reason text for an import |
| `move-entity` | Update source path after file rename/move |
| `remove-import` | Remove an import relationship |
| `remove-shared` | Unregister a shared entity |
| `remove-entity` | Remove entity and all references |
| `get-entity` | Get full entity snapshot (description, imports, exports) |
| `get-shared` | Get public API of an entity |
| `get-path` | Find shortest path between two entities |
| `get-orphans` | Find entities with no importers and not in any TOC |
| `read-toc` | Read the table of contents |

## Auto-Bootstrap Script

The script at `scripts/dsp_auto_bootstrap.py` rebuilds the entire DSP graph from scratch by scanning the codebase.

### Usage

```bash
# Full rebuild (always wipe first!)
rm -rf .dsp && python3 dsp-cli.py --root . init && python3 scripts/dsp_auto_bootstrap.py

# Preview without writing
python3 scripts/dsp_auto_bootstrap.py --dry-run
```

### What It Does

The bootstrap runs in 5 phases:

1. **Phase 1 -- Discover source files**: Walks `backend/`, `frontend/`, `ml/`, `workers/` directories. Excludes `node_modules`, `__pycache__`, test files, migrations, `conftest.py`, `__init__.py`.

2. **Phase 2 -- Parse imports**: Uses Python AST for `.py` files and regex for `.ts`/`.tsx` files. Resolves internal imports to project-relative paths. Identifies external packages (fastapi, react, etc.).

3. **Phase 3 -- Create entities**: Creates DSP object entities for each discovered source file. Generates purpose descriptions from docstrings/comments or path-based heuristics (e.g., `services/` -> "Business logic service"). Creates external dependency entities.

4. **Phase 4 -- Create import relationships**: Links entities via `add_import()` with auto-generated "why" text based on imported names (e.g., "Uses PriceService, PriceResponse + 2 more").

5. **Phase 5 -- Save uid_map.json**: Writes the source-path-to-UID mapping for downstream lookups.

### Performance

The script imports the `dsp-cli.py` Engine/Store classes directly (via `importlib.util`) instead of spawning subprocesses, completing a full rebuild in approximately 1.2 seconds.

### Configuration

Key constants in the script:

| Constant | Purpose |
|----------|---------|
| `SCAN_DIRS` | Directories to scan: `backend`, `frontend`, `ml`, `workers` |
| `EXCLUDE_PATTERNS` | Patterns to skip: `node_modules`, `__pycache__`, tests, migrations |
| `PYTHON_EXTERNALS` | Known external Python packages (not tracked as project entities) |
| `TS_EXTERNALS` | Known external TypeScript packages |
| `TS_ALIAS` | `@/` maps to `frontend/` |
| `PURPOSE_HINTS` | Path-to-description heuristics (e.g., `api/v1/` -> "API route handler") |

## Best Practices

### Before Refactoring

Always check dependencies before modifying a file:

```bash
# 1. Find the entity UID
python3 dsp-cli.py --root . find-by-source backend/services/price_service.py

# 2. Check who imports this file
python3 dsp-cli.py --root . get-recipients obj-a1b2c3d4

# 3. Check what this file imports
python3 dsp-cli.py --root . get-children obj-a1b2c3d4 --depth 1
```

### After Major Changes

If you add, remove, or rename files:

```bash
rm -rf .dsp && python3 dsp-cli.py --root . init && python3 scripts/dsp_auto_bootstrap.py
```

### Cycle Detection

Run periodically to catch accidental circular dependencies:

```bash
python3 dsp-cli.py --root . detect-cycles
```

The project has 1 known cycle: `alert_service` <-> `alert_renderer`. New cycles should be investigated and resolved.

## Limitations

- **Test files excluded**: The graph only contains source files, not test files
- **`__init__.py` excluded**: Package init files are not tracked as entities
- **Migrations excluded**: SQL migration files are not tracked
- **TypeScript parsing**: Uses regex (not a full parser), so complex import patterns may be missed
- **Append-only**: The bootstrap script adds entities without removing existing ones -- always wipe before rebuilding
