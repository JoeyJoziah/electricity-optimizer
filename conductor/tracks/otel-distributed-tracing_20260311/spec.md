## Summary

Add custom OpenTelemetry spans to critical backend services and connect trace export to Grafana Cloud, completing the distributed tracing story that the existing `observability.py` SDK foundation enables.

## Context

The OTel SDK foundation is **already implemented** (305-line `backend/observability.py`):
- TracerProvider with exporter selection (OTLP/Console/Silent)
- Auto-instrumentors: FastAPI, SQLAlchemy, httpx
- Request hooks injecting `user_id` and `request_id` on every span
- Settings: `OTEL_ENABLED` (bool), `OTEL_EXPORTER_OTLP_ENDPOINT` (str)
- App factory wiring in `create_app()` (guarded by `settings.otel_enabled`)
- 27 passing tests in `test_observability.py`
- Dependencies already in `requirements.txt`

However, **zero custom spans** exist across 26 backend services. The `get_tracer()` function is only called inside `observability.py` tests — no service code instruments business-critical operations. There is no Grafana Cloud account or OTLP export configured for production.

## User Story

As an **SRE/developer**, I want distributed traces for ML inference, API calls, external integrations, and business-critical operations so that I can diagnose latency bottlenecks, track error propagation, and monitor service health in Grafana Cloud.

## Technical Notes

- Use `get_tracer(__name__)` pattern in each service module
- Wrap critical methods with `with tracer.start_as_current_span("operation.name"):` blocks
- Add span attributes via `span.set_attribute("key", value)` — follow OTel semantic conventions
- Span naming convention: `{service}.{operation}` (e.g., `ml.predict`, `stripe.create_checkout`)
- Error recording: `span.set_status(StatusCode.ERROR, str(exc))` + `span.record_exception(exc)`
- All custom spans must be guarded by the existing `is_recording()` check pattern
- Grafana Cloud free tier: 50GB traces/month, 14-day retention — sufficient for RateShift traffic
